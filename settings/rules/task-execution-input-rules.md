# 任务执行输入识别规则

本文档定义了执行任务时从用户输入中识别需求、文件和任务范围的标准规则。

## 规则说明

当需要执行代码实现任务时，必须严格按照以下三步识别规则执行。

## 第一步：提取需求编号

用户必须提供需求编号，如002、002-XXX，搜索 `specs/002-*/` 以确定需求精确的目录，令 `需求目录` = 需求精确的目录

## 第二步：识别代码实现方案和任务清单文件

在 `需求目录` 下查找匹配的文件：

### 场景1：用户明确指定了模块或关键词

- 用户输入包含模块名称（如"用户服务"、"订单服务"）或其他关键词
- 查找规则：
  - 代码实现方案：`代码实现方案-*关键词*.md`
  - 任务清单：`任务清单-*关键词*.md`
- 输出：`已识别文件: 代码实现方案-XXX.md, 任务清单-XXX.md`

### 场景2：用户未指定模块，目录下只有一组文件

- 检查 `需求目录` 下是否只存在一个 `代码实现方案-*.md` 和对应的 `任务清单-*.md`
- 如果只有一组，直接使用该组文件
- 输出：`已识别文件: 代码实现方案-XXX.md, 任务清单-XXX.md`

### 场景3：用户未指定模块，目录下有多组文件

- 列出所有可用的代码实现方案和任务清单文件
- 终止流程，提示用户明确指定要执行的模块
- 输出：`发现多组文件，请指定模块: [列出所有文件组]`

## 第三步：识别任务范围

根据用户输入确定要执行的任务范围（严格按以下顺序判断）：

### 情况1：用户明确指定了任务编号

- 用户输入包含具体任务编号（如 `T001`、`T002`、`T001-T010`）
- 令 `执行范围` = 指定的任务清单
- 输出：`已识别执行范围: 任务 [T001, T002, ...]`

### 情况2：用户明确指定了阶段

- 用户输入包含阶段编号（如 `阶段1`、`阶段2`、`Phase 1`）
- 令 `执行范围` = 指定阶段的所有任务
- 输出：`已识别执行范围: 阶段X 的所有任务`

### 情况3：用户未指定任务或阶段

- 用户只提供了需求编号和可能的模块名称
- 令 `执行范围` = 任务清单中的所有任务
- 输出：`已识别执行范围: 全部任务`

## 验证和确认

完成三步识别后，必须输出识别结果摘要：

```
【识别结果】
- 需求目录: specs/XXX/
- 代码实现方案: 代码实现方案-XXX.md
- 任务清单: 任务清单-XXX.md
- 执行范围: [全部任务 / 阶段X / 任务T001,T002,...]
```

## 任务解析规则

根据第三步识别的 `执行范围`，从任务清单中提取对应的任务：

### A. 执行范围 = 全部任务

- 识别任务清单中的所有开发阶段
- 提取每个阶段的所有任务项

### B. 执行范围 = 阶段X

- 定位到指定的阶段X
- 提取该阶段的所有任务项
- 检查该阶段的前置依赖阶段是否已完成

### C. 执行范围 = 任务[T001, T002, ...]

- 定位到指定的任务编号
- 提取这些任务的详细信息
- 检查这些任务的前置依赖任务是否已完成

### 提取内容（针对要执行的任务）

1. **阶段划分**：识别相关的开发阶段(如：基础设施、模型层、业务逻辑层、接口层等)
2. **任务列表**：每个阶段的具体任务项，格式为 `- [ ] [TaskID] [P?] [代码实现方案编号?] [结构化需求编号?] 任务描述（包含文件路径）`
3. **依赖关系**：
   - 阶段间依赖：哪些阶段必须串行，哪些可以并行
   - 任务间依赖：标记`[P]`的任务可并行执行
   - 如果执行部分任务，检查依赖的任务是否已完成（标记为`- [x]`）
4. **文件路径映射**：提取每个任务对应的Java文件完整路径
5. **章节引用**：任务引用的代码实现方案章节号(如[2.4.3])

### 依赖性检查

- 如果要执行的任务依赖于未完成的任务，列出依赖清单并提示用户
- 示例输出：`警告：任务T010依赖于未完成的任务T003、T005，建议先完成依赖任务`
