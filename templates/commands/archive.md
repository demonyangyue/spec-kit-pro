## Core Task

归档已完成的需求，将需求文档和代码变更内容提取并更新到应用知识库（`.specify/memory/`），建立规范化的需求闭环追溯链路。

## Execution Steps

创建 TODO 任务项，按以下步骤执行：

### 0. 前置校验（**必须通过**）

执行前必须满足：
- 在**项目根目录**执行本命令
- 存在 `specs/` 目录；若不存在，提示："未找到 specs/ 目录，请先创建需求目录或执行 specify init 初始化项目"，终止流程
- 根据用户输入（需求编号或完整目录名）在 `specs/` 下能**唯一匹配**到目标需求目录（如 `specs/002-*` 或 `specs/001-用户认证功能`）；若匹配不到或匹配到多个，提示并终止

### 1. 需求识别与验证

**识别需求目录**
- 用户可能提供需求编号（如 001, 002）或完整目录名（如 001-用户认证功能）
- 搜索 `specs/` 下匹配的需求目录：`specs/[编号]-*/` 或完全匹配 `specs/[目录名]`
- 验证需求目录存在，包含必要文件

**验证完成状态**

1. **检查必要文件**
   - 检查 `specs/[需求目录]/技术方案.md` 是否存在
   - 如果不存在，终止归档流程

2. **检查任务清单**
   - 先搜索 `specs/[需求目录]/任务清单-*.md` 文件；若不存在，再查找 `specs/[需求目录]/tasks.md`
   - 如果存在任务清单文件：
     - 读取任务清单内容
     - 统计任务状态：
       - 已完成任务数量：`- [x]` 或 `- [X]`
       - 未完成任务数量：`- [ ]`
     - 如果存在未完成任务：
       - 显示未完成任务列表
       - 询问用户："检测到 N 个未完成任务，是否继续归档？"
       - 如果用户拒绝，终止归档流程
   - 如果不存在任务清单文件：
     - 提示："未找到任务清单文件（任务清单-*.md 或 tasks.md），无法验证任务完成情况"
     - 询问用户："是否继续归档？"
     - 如果用户拒绝，终止归档流程

3. **检查代码提交状态**
   - 使用 git 检查未提交的变更：
     ```bash
     git status --short
     ```
   - 如有未提交代码：
     - 显示未提交文件列表
     - 提示用户："请先提交代码再执行归档"
     - 终止归档流程

4. **分析相关提交**
   - 获取需求相关的提交记录：
     ```bash
     git log --oneline --since="[需求目录创建时间]" --pretty=format:"%h %s" | head -20
     ```

5. **用户最终确认**
   - 询问用户确认需求是否已完成并部署
   - 如果用户确认未完成，终止归档流程

### 2. 分析需求内容

**读取需求文档**
- 读取 `specs/[需求目录]/结构化需求.md`（如存在）
- 读取 `specs/[需求目录]/技术方案.md`
- 读取 `specs/[需求目录]/Web接口设计.md`（如存在）
- 读取 `specs/[需求目录]/需求与代码库的Gap分析.md`（如存在）

**提取归档信息**

从需求文档中提取以下内容：

| 归档维度 | 提取内容 | 来源文档 |
|---------|---------|---------|
| **产品知识** | • 新增业务实体及关系<br>• 新增业务术语<br>• 新增业务状态类型<br>• 新增业务流程 | 结构化需求.md<br>技术方案.md |
| **技术架构** | • 新增外部系统集成<br>• 新增中间件&基础产品封装<br>• 新增工具类方法<br>• 新增业务服务封装<br>• 新增架构模式 | 技术方案.md<br>Web接口设计.md |
| **存储模型** | • 新增数据表<br>• 修改数据表结构 | 技术方案.md |

**分析代码变更**
- 使用 git diff 分析需求相关的代码变更：
  ```bash
  # 获取需求目录创建后到当前的代码变更
  git diff [需求目录创建时的commit] HEAD --stat

  # 查看新增的文件
  git diff [需求目录创建时的commit] HEAD --name-status | grep "^A"
  ```
- 识别新增的关键文件（实体类、服务类、工具类、API接口等）

### 3. 更新产品知识

**目标文件**：
- `.specify/memory/product/领域模型和术语.md`
- `.specify/memory/product/存储模型索引.md`

**更新原则**
- **增量更新**：只添加新增内容，不删除已有内容
- **去重合并**：如实体已存在，补充新属性；如术语已存在，丰富定义
- **保持结构**：遵循现有文档的章节结构和格式

**领域模型和术语更新**

检查是否有以下内容需要归档：

1. **新增实体**
   - 实体名称（中文 + 英文）
   - 实体定义（1-2句话）
   - 核心属性（3-5个关键属性）
   - 示例：
     ```markdown
     **订单退款记录 (OrderRefundRecord)**
     - **定义**：记录订单退款的申请、审批、执行全流程
     - **核心属性**：退款单号、关联订单、退款金额、退款状态、审批人
     ```

2. **实体关系**
   - 如新增实体与已有实体有关联，更新 Mermaid ER 图
   - 标注关系类型（1:1, 1:N, N:M）

3. **业务状态**
   - 新增状态枚举及流转规则
   - 示例：
     ```markdown
     **退款状态 (RefundStatus)**
     - PENDING(待审批) → APPROVED(已批准) → COMPLETED(已退款)
     - PENDING(待审批) → REJECTED(已拒绝)
     ```

4. **业务术语**
   - 新增领域特定术语及定义
   - 示例：
     ```markdown
     | 术语 | 定义 | 备注 |
     |-----|------|------|
     | SPU | Standard Product Unit，标准产品单元 | 描述商品的抽象信息 |
     ```

**存储模型索引更新**

检查是否有新增或修改的数据表：

1. **新增表**
   - 添加到表清单，格式：
     ```markdown
     | 序号 | 表名 | 表说明 | 关键字段 |
     |-----|------|--------|---------|
     | N | order_refund | 订单退款记录表 | id, order_id, refund_amount, status |
     ```

2. **表结构变更**
   - 如果是重要字段新增（非扩展字段），在备注中说明
   - 不记录每次字段变更的详细历史，只保持当前状态

## Important Constraints

### 归档原则

| 原则 | 说明 |
|-----|------|
| ✅ **增量更新** | 只添加新内容，不删除已有知识 |
| 🎯 **精准提取** | 只归档通用可复用的知识，避免过度归档 |
| 📐 **结构一致** | 严格遵循现有文档的格式和章节结构 |
| 🔍 **去重合并** | 检查重复内容，合并而非新增 |
| ⚖️ **平衡详略** | 保持适度详细，避免冗余和遗漏 |
| 🚫 **不修改公共知识** | 不修改 `.specify/memory/common/` 下的文件 |
| 🚫 **不修改用户自定义** | 不修改用户手动添加的自定义文档（如 custom、自定义等） |

### 归档范围

**需要归档的内容**：
- 新增的领域实体、术语、状态、流程
- 新增的数据表（但不记录详细变更历史）
- 新增的外部系统/中间件&基础产品封装
- 新增的通用工具类和业务服务
- 新引入的架构模式

**不归档的内容**：
- 需求特定的临时逻辑（不可复用）
- 单次使用的查询或脚本
- 业务配置和参数（会频繁变化）
- 具体的实现细节（由代码承载）
- 每次的表字段变更历史

### 更新策略

**产品知识**：
- 领域模型：增量补充实体和关系
- 术语表：追加新术语，丰富已有术语定义
- 存储模型：维护表清单的当前状态

**技术架构**：
- 集成能力：记录新增集成点和使用场景
- 工具服务：记录可复用的工具和服务方法
- 架构模式：记录新引入的通用架构模式

**规范文档**：
- 不在归档时更新规范文档（规范由团队统一维护）

## Safety & Fallback

**需求目录不存在**：
- 终止流程
- 提示：`specs/` 下未找到需求目录 `[需求编号]-*` 或匹配不唯一
- 建议：检查需求编号是否正确，或先创建 specs 目录

**任务清单未完成**：
- 检测到任务清单中有未完成任务
- 显示未完成任务列表
- 询问用户："检测到 N 个未完成任务，是否继续归档？"
- 如用户拒绝，终止归档流程

**缺少任务清单**：
- 未找到 `任务清单-*.md` 或 `tasks.md` 文件
- 提示：无法验证任务完成情况
- 询问用户："是否继续归档？"
- 如用户拒绝，终止归档流程

**需求未完成**：
- 询问用户是否确认完成
- 如用户确认未完成，终止归档
- 提示：归档仅适用于已完成并部署的需求

**代码未提交**：
- 检测到未提交的代码变更
- 提示：请先提交代码变更，再执行归档
- 建议：`git add . && git commit -m "[需求编号] 实现说明"`

**无可归档内容**：
- 如分析后发现没有需要归档的知识
- 仅移动需求目录到归档，不更新知识文件
- 报告：本次需求无需更新应用知识库

**重要文件不存在**：
- 如 `.specify/memory/AGENTS.md` 不存在
- 提示：应用知识库未初始化
- 建议：先执行 `specify init` 或使用 `/speckit.constitution` 命令初始化项目与知识库
